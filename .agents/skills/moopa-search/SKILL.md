# Moopa Search with pg_trgm

Moopa uses PostgreSQL's pg_trgm extension for fuzzy text search via trigram similarity. This provides typo-tolerant, partial-match search that works in any PostgreSQL environment.

## How It Works

1. Mark fields as `searchable: true` in table definitions
2. Moopa auto-generates a `search_text` column concatenating all searchable fields
3. A GIN index with `gin_trgm_ops` enables fast similarity matching
4. Search queries use the `<%` operator with `word_similarity()` for ranking

## Making Fields Searchable

In your table definition CFC, add `"searchable": true` to any field:

```cfml
"fields": {
    "full_name": {
        "type": "varchar",
        "searchable": true
    },
    "email": {
        "type": "varchar", 
        "searchable": true
    },
    "company_name": {
        "type": "varchar",
        "searchable": true
    }
}
```

## Searching JSONB Fields

JSONB fields cannot be directly searched. Create a generated column that extracts the searchable text:

```cfml
"address": {
    "type": "jsonb",
    "html": { "control": "address" }
},
"formatted_address": {
    "label": "Formatted Address",
    "type": "text",
    "generation_expression": "address->>'formatted_address'",
    "searchable": true,
    "html": { "hidden": true }
}
```

The framework automatically inlines the `generation_expression` into the `search_text` column (avoiding PostgreSQL's restriction on generated columns referencing other generated columns).

## Generated Schema

For a table with searchable fields, Moopa generates:

```sql
-- Generated column concatenating all searchable text
ALTER TABLE my_table ADD COLUMN search_text text 
GENERATED ALWAYS AS (
    COALESCE(full_name::text, '') || ' ' || 
    COALESCE(email::text, '') || ' ' || 
    COALESCE((address->>'formatted_address')::text, '')
) STORED;

-- GIN index for fast trigram matching
CREATE INDEX my_table_search_trgm_idx 
ON my_table USING GIN (search_text gin_trgm_ops);
```

## Using Search in Routes

### Basic Search with db.search()

```cfml
<cffunction name="search">
    <cfset searchTerm = request.data.filter.term ?: '' />
    
    <cfreturn application.lib.db.search(
        table_name = "my_table",
        field_list = "id,full_name,email",
        q = searchTerm,
        limit = 100
    ) />
</cffunction>
```

The `q` parameter triggers trigram search when the table has searchable fields.

### Get IDs Only with idsInSearchTerm()

```cfml
<cfset matchingIds = application.lib.db.idsInSearchTerm(
    table_name = "my_table",
    term = "search query",
    limit = 20
) />
```

Returns an array of matching IDs ordered by relevance.

## SQL Generated by Search

When you call `db.search()` with a `q` parameter:

```sql
SELECT ...
FROM my_table
WHERE 'search term' <% search_text
ORDER BY word_similarity('search term', search_text) DESC
LIMIT 100
```

- `<%` operator: Returns rows where the query has word similarity to the text
- `word_similarity()`: Scores how similar the query is to words in the text (0-1)
- Results ordered by relevance score descending

## Frontend Integration

Typical Alpine.js search pattern:

```html
<input type="text" 
       x-model="filters.term" 
       @input.debounce.500ms="load()" 
       placeholder="Search...">
```

```javascript
async load() {
    this.records = await req({
        endpoint: 'search',
        body: { filter: this.filters }
    });
}
```

## Requirements

The `pg_trgm` extension must be enabled:

```sql
CREATE EXTENSION IF NOT EXISTS pg_trgm;
```

This is included in all standard PostgreSQL distributions.

## Comparison with ILIKE

| Feature | pg_trgm (`<%`) | ILIKE |
|---------|----------------|-------|
| Typo tolerance | Yes | No |
| Partial word matching | Yes | Yes (with %) |
| Word boundary awareness | Yes | No |
| Relevance ranking | Yes | No |
| Index support | GIN index | Limited |
| Performance on large tables | Good | Poor |

## Tips

- Keep searchable fields to essential text columns (name, email, etc.)
- For JSONB, always create a generated column to extract searchable text
- The `search_text` column is auto-excluded from query results
- Schema sync automatically creates/updates the index
